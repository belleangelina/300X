# 300X 的 AI Agent 开发与交付闭环

300X 是一个使用 Flutter 开发的百合会论坛第三方阅读器，维护 Android、Linux 和 iOS 三个平台工程，已实现登录、目录浏览、作品聚合、漫画与小说阅读、云收藏、本机历史和离线下载。

项目的大部分代码实现、测试补充、问题排查和工程文档由 AI Agent 完成。我负责确定产品目标、技术边界和验收标准，提供实际运行反馈，在关键方案之间作出取舍，并决定功能是否可以进入发布版本。整个开发过程由多轮需求拆解、编码、验证、定位和复验组成，不是通过一段提示一次生成完整应用。

本文只保留适合公开的工程过程。

## 项目来源与新增范围

300X 基于 GPL-3.0 项目 [xiaoyaocz/flutter_dmzj](https://github.com/xiaoyaocz/flutter_dmzj) 进行二次开发，界面结构和主要交互参考上游项目。300X 的新增工作主要在论坛数据源和新的业务链路，上游已有的界面基础不计为本项目原创内容。

项目重新建立了 Flutter 工程和业务分层，并围绕论坛数据源实现了新的核心链路，主要包括：

- Discuz 登录表单、图片验证码、Cookie 会话恢复，以及 Android/iOS WebView 登录兜底；
- 漫画区、轻小说和文学区的列表、分类、排序、分页与搜索；
- 论坛主题、楼层、Tag 和跨帖链接解析，以及作品、来源目录和章节的高置信聚合；
- 漫画与小说阅读器、本机历史、阅读锚点、云收藏和离线下载；
- SQLite 索引、认证媒体缓存、缓存维护和下载任务状态机；
- Android、Linux 和 iOS 的条件实现、构建脚本与 GitHub Actions 发布流程。

实际开发中，最难处理的是不稳定的论坛数据。标题、目录格式和历史帖子结构并不统一，同一作品可能由不同用户发布，章节也可能分散在同帖楼层、多个主题或 Tag 分页中。300X 的解析策略优先保证准确：证据不足时宁可保留多个作品或单章，也不根据标题相似度强行合并。

## Agent 如何参与开发

项目以设计规格为功能和边界的统一依据。每轮开发先写清用户路径、数据边界、失败状态和验收条件，再由 Agent 定位代码、实现修改并补充定向测试。完成代码后，还要继续走完下面的流程：

```text
需求与验收条件
    -> Agent 定位并实现
    -> 静态检查与定向测试
    -> 目标平台构建
    -> 执行真实功能路径
    -> 保存脱敏证据
    -> 定位失败原因
    -> 修复并重跑原路径
    -> 回归相邻功能
```

Agent 主要承担大范围代码检索、方案比较、实现、测试生成、日志分析和重复验证。产品边界仍由我确定，“测试通过”也不能直接等同于“用户路径可用”。例如，论坛写操作是否允许自动重试、不同楼主的同名作品能否合并、下载失败后保留哪些文件，都需要提前给出明确规则。

项目用几种状态记录每轮进度：

- 代码已经实现，但没有完成目标平台运行，只能标记为“实现中”；
- 自动化和构建通过，但等待真实设备操作，标记为“待实机”；
- 修复已经完成，但尚未重跑失败路径，标记为“复验中”；
- 只有原路径、相邻功能和证据记录都完成后，才能标记为“通过”；
- 外部条件没有自然出现，例如验证码或特定风控流程，明确标记为“受限”，不冒充已完成验证。

这些状态用来防止过早宣布完成。Agent 可以很快写出实现，但最终状态仍由目标平台上的实际结果决定。

## 案例一：长目录从一分钟以上恢复到可用

一次性能回归中，某个包含 422 个章节的小说首次进入时间从十几秒增长到一分钟以上，有时最终只显示一个章节。最初怀疑是 HTML 解析太慢，也考虑过把解析直接移入后台 isolate。

采样结果排除了这个判断：三份 100～146 KB 的真实 HTML 在 AOT 环境下完成 DOM 解析只需要约 34～49 ms，章节归并也处于毫秒级。主要耗时来自网络请求和错误的控制流程。

继续沿真实路径追踪后发现两个问题：

1. 后台章节解析闭包捕获了界面状态和进度回调，其中包含不能跨 isolate 发送的对象。网络请求结束后解析任务失败，程序随后退回单章目录。
2. 首楼已经包含完整章节链接，但旧流程仍串行读取数十页楼主内容；封面补探测又可能占用同一个共享请求，使目录解析被无关任务拖住。

修复重点放在控制流程和任务所有权：

- isolate 任务只携带可发送的纯数据，不捕获界面状态、数据库、网络会话和取消令牌；
- 章节结果回到主 isolate 后再检查页面是否仍然有效；
- 对首楼已经具备可靠完整目录的小说，只读取第一页建索；
- 仅复用已经完成的全帖缓存，不等待正在执行的封面探测；
- 建立目录期间暂停自动封面队列，并限制补探测页数。

修复后在全新 Linux 数据目录中执行冷路径，从进入作品到显示 422 章耗时 13.675 秒，持久索引条数和首尾章节均正确。剩余时间主要来自论坛请求，不再把网络墙钟时间误报为本地解析开销。相关失败路径随后加入回归测试，并重新执行静态检查和真实目录验证。

## 案例二：用 Profile 定位封面滚动卡顿

主页冷缓存加载时曾出现明显卡顿。最初的候选原因包括列表一次构建过多、动画优先级不足、标题聚合开销和网络并发过高。如果没有采样，任何一个方向都可能产生看似合理但无效的修改。

检查真实缓存后发现，374 张封面平均约 277 万像素，其中 48 张超过 400 万像素，最大图片达到 4319×6084。列表中一张封面实际只显示约 80×110 逻辑像素，但旧路径会按原图完成验证和解码。

同一网格滚动场景中，旧显示路径的 Flutter 图片缓存占用约 103 MB，只容纳了 12 张普通缓存图片；按实际物理显示尺寸解码后，占用降至约 6.53 MB，并可容纳 64 张图片，下降约 94%。

最后做了三项改动：

- 原始文件继续落盘，验证阶段只解码最长边不超过 256 px 的帧，显示阶段按实际物理尺寸解码；
- 建立封面加载协调器，对可见需求计数，限制并发，并在触摸和惯性滚动期间暂停新的解析与下载阶段；
- 隐藏页面不再发起封面任务，离屏条目可以取消尚未开始的后续工作，但不强行中断已经进入共享网络层或 Flutter 解码器的请求。

与此同时，论坛 HTML、搜索结果和章节解析等纯 CPU 工作被迁移到后台 isolate，网络会话、Cookie、数据库和页面状态仍留在主 isolate。

在包含 422 章目录刷新的 Profile 时间线中，UI 帧最大耗时为 13.069 ms，Raster 帧最大耗时为 1.501 ms；持续拖动时不再有新的封面文件落盘，停止滚动后队列恢复。后续采样还发现 Linux 首次排版混合中日文标题时存在字体回退尖峰，因此只对 Linux 设置系统字体候选，没有把平台特有结论直接套用到 Android。

这次排查先用 Profile 排除了错误方向，再收敛到影响范围较小的修改。Linux 上得到的字体结论没有直接外推到 Android，仍然等待对应平台的采样结果。

## 案例三：Android 真机协作闭环

Linux 环境可以完成主要开发和首轮验证，但触摸、安全区、系统栏、覆盖安装和移动网络行为仍需 Android 真机确认。Agent 负责构建 APK、整理检查步骤和分析日志，我在 Android 16 真机上安装、操作并回传结果。发现问题后，再由 Agent 修改和重建，随后复验原路径。

真机上检查了：

- 首次安装、覆盖安装、冷启动和登录会话恢复；
- 漫画、小说、分类、搜索冷却和云收藏；
- 漫画双向分页、小说分页、操作反转、沉浸式系统栏和阅读进度恢复；
- 下载暂停、继续、完成、应用重启恢复以及飞行模式下的离线阅读；
- 测试结束后恢复临时收藏、网络和本机设置，避免把验证副作用留给用户环境。

真机发现了桌面和 Widget 测试没有暴露的安全区问题：顶部标签在高密度窄屏设备上受到系统区域挤压。修复后重新核对标签高度、系统返回和冷启动路径。验证过程中没有故意连续提交错误密码，因此验证码和 WebView 风控入口没有自然触发；这些路径只保留自动化与产物检查，不宣称已经完成真实风控场景。

真机协作时，Agent 只给出必要的操作步骤和证据要求，我负责必须依赖实体设备的动作。回传结果后继续定位、重建和复验，直到原问题关闭。

## 发布门禁

日常修改只运行受影响的静态检查、单元测试和 Widget 测试，避免每次变更都执行耗时的全量构建；进入 Release 时再执行完整门禁。当前公开版本包含 49 个测试文件，按 `test` 和 `testWidgets` 声明统计约 300 个自动化用例。

发布脚本统一处理版本递增、失败回滚、Android 通用与 arm64 APK、Linux bundle 和产物命名。GitHub Actions 在 macOS runner 上构建未签名 iOS IPA，并与 Android 产物一起附加到 Release。由于当前没有可用的 macOS 和 iOS 实机环境，项目只声明 iOS 工程与 CI 构建完成，不声明 iOS 实机闭环。

发布脚本也经历过实际故障。一次 Android Release 构建运行数分钟后，Gradle daemon 异常退出。内核日志显示宿主物理内存不足，而项目配置的 Java 堆和元空间上限超过了机器容量。收敛 Gradle 内存配置后重新执行双平台构建，并复核 APK 版本、ABI、签名和哈希。

## 公开仓库与内部证据

项目早期在私有工作区通过 AI Agent 高频迭代，期间产生了大量实验性修改、临时提示、失败日志和包含受限内容的验证材料。达到可用初版后，我在公开开源前整理了提交历史和文档，因此 GitHub 历史从一个可运行的版本快照开始。

完整内部台账仍然保留，用于记录每个功能的状态、失败现象、根因、修复、复验和证据位置；公开仓库只发布脱敏后的规格、工程复盘、测试和版本记录。从首个公开版本开始，缺陷修复、多端构建和 Release 流程保留正常的增量记录。

两类材料分开保存：

- 内部材料用于追溯和复验，可以保留完整细节，但必须保护账号、论坛受限内容和本机环境；
- 公开材料只保留理解项目所需的方法、边界和结果，不带入无关的私有信息。

## 小结

这次开发让我确认，Agent 能够承担大量代码检索、实现、测试和重复验证工作，但前提是需求边界清楚、结果可以观察、失败路径能够复现。产品取舍、风险判断、实机反馈和最终验收仍由我负责。

相比单纯提高编码速度，我更看重这套流程能否把需求、证据和修复串起来，最后交付一个可以安装、运行、回归和持续发布的版本。
