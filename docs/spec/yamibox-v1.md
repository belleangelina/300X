# 300X v1 设计规格

状态：已批准
日期：2026-07-10
目标版本：v1.0.0

功能只有完成“实现、测试、目标平台构建、实际运行、缺陷修复、重新构建与回归”的闭环后才算完成。首个完整闭环在 Linux 执行；Android 实机阶段由用户协助操作和回传证据；iOS 在具备 Apple 工具链后补验。

## 1. 项目目标

300X 是一个面向个人使用、计划公开发布到 GitHub 的 Flutter 阅读器。应用从百合会论坛中读取用户有权访问的漫画和小说帖子，将论坛主题、楼层与跨帖链接整理成适合阅读器使用的作品和章节，并复刻 `xiaoyaocz/flutter_dmzj` 当前 `main`/v2.1.2 中保留功能的 UI 与交互。

“完美复刻”的范围定义如下：

- 已确认保留的功能，尽量逐项复刻参考应用的页面结构、视觉层级、尺寸、明暗主题、动效和交互。
- 已确认删除的功能不出现在应用中，也不保留空入口。
- 因论坛数据模型与动漫之家 API 不同而新增的来源筛选、来源标签、文字封面和原帖兜底属于允许差异。
- 不复用“动漫之家X”的名称和 Logo；300X 使用独立品牌。
- 不承诺不同操作系统字体栅格化、系统状态栏和原生控件达到逐像素一致，但应用自身组件需要保持一致。

## 5. 信息架构

### 5.1 主导航

窄屏底部导航包含三个无文字主入口：

1. 漫画
2. 小说
3. 我的

宽屏使用与参考应用一致的左侧 NavigationRail。列表区域保持约 450 px 宽，详情在右侧子导航区域显示；阅读器使用独立全屏路由。

### 5.2 漫画

漫画来源固定为：

`https://bbs.yamibo.com/forum.php?mod=forumdisplay&fid=30&mobile=2`

漫画主页只显示“漫画区”，不再划分推荐、更新、分类和排行页签。列表上方按“列表/网格、跳页、主题分类、最新/热度”排列四个等宽触发区，不使用圆角面板，默认使用“列表 + 第 1 页 + 全部 + 最新”。分类使用贴近触发区展开的锚定菜单；最新/热度只有两种状态，点击后直接切换。分类与排序组合为同一个论坛目录请求，分页沿用相同条件；网格从手机三列起步并随宽度自适应。

### 5.3 小说

小说来源：

- 文学区：fid=49
- 轻小说（论坛轻小说/译文区）：fid=55

小说主页使用“轻小说”和“文学区”两个独立页签，按此前后顺序排列，不提供混合“全部”页签。每个页签独立维护列表/网格视图、分类、最新/热度排序、分页起点、分页游标和当前滚动位置；默认均为“列表 + 第 1 页 + 全部 + 最新”。搜索仍一次限制 fid=49 和 fid=55，收藏、历史和下载不按版块拆分。

漫画和小说当前主入口被再次点击时，当前页签先滚动到顶部，再刷新当前条件和当前分页起点；下拉刷新则回到第 1 页并刷新。跳页直接替换为目标页并回到顶部；继续滚动加载后显示已加载页范围，分类或排序变化时回到第 1 页。作品列表已经由入口确定版块，因此不重复显示楼主和版块标签；分类与章节信息合并成一行，为标题保留更多空间。列表/网格、跳页、分类和排序的控制条保持单行，不追加重复说明或装饰图标。

### 5.4 我的

“我的”页沿用参考应用的卡片布局，包含：

- 账号信息、论坛头像、退出和更换账号
- 云端收藏
- 本机阅读历史
- 漫画下载
- 小说下载
- 显示主题
- 更多设置
- 开源主页
- 关于APP（含免责声明）

不显示评论、云端历史、本地收藏、检查更新等已删除入口。

账号信息下的功能入口分为两组：第一组包含云端收藏、本机阅读历史和两类下载，第二组包含显示主题、更多设置、开源主页和关于 APP。“显示主题”不重复标注当前生效模式，配置入口保持同高。浅色模式使用浅灰页面底色和白色分组卡片，深色模式使用深色页面底色和稍亮卡片，两组信息层级保持一致。“开源主页”是应用内唯一 GitHub 入口。更多设置顶部只显示页面标题，“常规 / 漫画 / 小说 / 下载”使用底部等宽纯文字栏，不附加图标。

## 6. 数据模型

### 6.1 核心实体

`SourceThread` 表示一个论坛主题：

- tid
- fid、typeId 和来源名称
- 原始标题、摘要、作者/发帖人、头像
- 发布时间、最后回复时间
- 浏览数、回复数和热度信息
- 分页和原帖 URL
- 首帖及解析出的楼层引用

`SourcePost` 表示一个楼层：

- pid、tid、页码和楼层顺序
- 发帖用户、时间
- 清洗后的正文块
- 图片、附件和链接
- 是否为楼主/目录目标

`Work` 表示 APP 中的作品：

- 本地稳定作品 ID
- 漫画或小说类型
- 规范化标题、显示标题、作者信息和版本标记
- 来源版块、分类、简介、封面
- 一个或多个来源主题
- 一个或多个按论坛楼主分组的来源目录
- 云端收藏映射

`WorkDirectory` 表示同一作品的一份独立来源目录：

- 稳定目录 ID 和论坛楼主
- 该楼主发布或明确归属到该目录的来源 tid
- 目录内有序章节列表

不同楼主发布同一作品时仍属于一个 `Work`。各独立来源目录自身不跨楼主覆盖或改写；智能目录可以按后述规则跨目录保守补缺，但不会改变独立目录。无法可靠判断楼主的 Tag/楼内链接在多目录作品中进入单独的“未归属来源”目录。

`Chapter` 表示阅读章节：

- 本地稳定章节 ID
- 显示标题和可排序章节标记
- 来源 tid/pid 或跨帖 URL
- 漫画图片列表，或小说结构化正文
- 更新时间和下载状态

其他本地实体：

- `ReadingProgress`
- `ReadingHistory`
- `CloudFavoriteCache`
- `DownloadTask`
- `DownloadedAsset`
- `CoverEntry` 与 `CoverAlias`
- `AppSetting`

### 6.2 稳定 ID

- 论坛来源标识始终保留 tid/pid，不以可变标题作为唯一键。
- 单帖作品优先使用 `forum-thread:{tid}`。
- 跨帖聚合作品使用规范化作品键的稳定摘要，保存全部来源 tid 及其目录归属。
- 聚合规则升级时，不静默丢弃已有历史；通过来源 tid/pid 重新关联。

## 7. 登录与会话

### 7.1 正常登录

首次启动由用户手动输入论坛账号和密码。开发目录外层的 `bbs` 文件只用于人工验证，绝不打包、复制或提交到 300X 仓库。

登录流程：

```text
读取登录页
    -> 解析 formhash、loginhash、referer 和安全提问
    -> 提交账号密码
    -> 解析成功、失败、验证码或其他挑战
    -> 成功后持久化 Cookie 并验证受限版块
```

- 账号和密码保存到各平台安全存储。
- Cookie 使用持久 CookieJar，并与账号绑定。
- 启动时先验证 Cookie；失效后才使用安全存储中的凭据重新登录。
- 启动校验遇到连接失败时：已有成功保存的凭据继续进入本地离线界面；没有凭据则进入登录页并显示中文网络提示，不得用初始化错误页阻断用户输入或后续重试。
- 日志必须过滤用户名、密码、Cookie、formhash 和带认证参数的 URL。
- 退出登录清除安全存储凭据、Cookie 和账号相关缓存；已下载内容和本地阅读历史默认保留。
- 切换账号后重新加载云端收藏，不能把前一账号收藏状态写入新账号。

### 7.2 验证码

已知论坛可能在连续登录失败后要求验证码。

- 每次失败后重新获取并解析登录表单，不盲目重复提交。
- 三端均支持 Discuz 图片验证码：使用同一 Cookie 会话加载验证码图片，并在原生登录界面输入。
- 验证码错误后刷新验证码、formhash 和 loginhash。
- 登录 POST 不做透明自动重试，防止一次用户操作被重复提交并加剧风控。

### 7.3 WebView 兜底

- Android 和 iOS 的原生登录页始终提供低强调的“网页登录”入口，不要求应用先准确识别验证码、JavaScript 挑战或论坛风控；是否进入 WebView 由用户决定。
- 登录成功后，从 WebView CookieStore 读取 `bbs.yamibo.com` Cookie，同步到应用网络层，再访问受限版块验证。
- 应用已有 Cookie 在打开 WebView 前写入 WebView CookieStore。
- Linux 不引入非官方或不稳定的 WebView 实现；Linux 依赖原生登录和图片验证码。

## 8. 网络访问约束

- 所有论坛请求使用同一个受控客户端、CookieJar 和明确 User-Agent。
- 论坛附件请求携带登录 Cookie、正确 Referer，并仅使用 HTTPS。
- 列表和详情采用低并发；封面探测并发上限应比普通页面请求更低。
- 论坛 HTML 的列表、帖子、Tag 目录和搜索解析，以及纯章节解析，必须在后台 isolate 执行；网络会话、Cookie、数据库和页面取消状态留在主 isolate。后台结果返回后再次检查取消状态，已退出详情不得继续提交旧目录结果。
- 仅幂等 GET 可在连接重置时有限重试，使用指数退避和随机抖动。
- 登录、收藏和取消收藏等写操作不自动重试。
- 尊重论坛权限、隐藏内容和访问限制，不绕过阅读权限、回复可见、积分或其他限制。
- 遇到权限内容时显示原因和“在浏览器打开原帖”，不伪装为空章节。
- 按目录 PID 打开章节时，以目标楼层和下一章 PID 作为正文边界，并在边界内按目标楼层作者收集续楼；不得依赖跳转落点页首楼推断楼主，否则后续分页会误丢正文。
- 权限提示与阅读器底栏的原帖入口都打开章节自身的精确来源地址；解析响应的分页地址只用于正文图片 Referer。
- HTML 结构无法识别时保留原帖入口，并记录不含正文和凭据的诊断信息。

## 9. HTML 解析

### 9.1 列表解析

从移动版列表解析：

- tid 和链接
- 标题、摘要和主题分类
- 发帖人、头像和时间
- 浏览数、回复数
- 置顶/公告状态
- 当前页、总页数和下一页
- 最新、热门、新帖、精华及 typeId 过滤链接

公告、版规、关联任务帖、问题反馈帖、发帖教程等行政主题不得进入主页最新/热度、分类筛选和搜索作品结果，除非未来有显式产品规则。行政判断同时读取置顶状态、标题和解析后的分类名；分类精确为 `#公告/公告` 时即使标题不含“公告”且未置顶也必须剔除，不依赖可能变化的 typeId。`#公告/公告` 分类本身也不得进入主页分类选择面板；带其他 typeId 的筛选请求返回行政主题时仍执行同一过滤。

### 9.2 帖子解析

- 以 `.plc.cl[id^=pid]` 和 `.message` 等稳定语义组合定位楼层，不依赖单一绝对 DOM 路径。
- 保留段落、换行、标题、强调、图片和必要链接。
- 排除头像、表情、论坛 Logo、签名、普通引用块、评分、点评和操作按钮。部分旧小说首帖会把整段正文错误包在 `quote/blockquote` 中：仅当该块位于楼主首帖、不含引用归属链接且清洗后正文不少于 600 字时，将其标记为实质正文并保留；回复楼层、漫画和短引用仍按引用排除。
- 漫画图片按正文 DOM 顺序排列。
- 小说正文转换为文本段落、标题和插图块，不把 HTML 原样交给 WebView 阅读。
- 相对链接统一解析为 `https://bbs.yamibo.com/` 下的绝对链接。
- 目录重定向链接解析 tid、pid 和目标页；不得依赖重定向后的临时 URL 作为唯一标识。

### 9.3 目录与章节

明确标记为 `#短篇漫畫/#短篇漫画` 的漫画默认是一帖一作品，但标签不是不可纠正的绝对短路条件。无论入口来自主页、排行、更新还是搜索，首次打开都加载当前主题并补齐真实分类；只有楼主正文或作品 Tag 中存在至少两个不同主线章号、链接指向同一规范作品的结构化目录证据时，才允许把误标短篇提升为长篇并进入统一跨帖解析。短篇集、关联作品、普通回复和仅标题相似的链接不能越过门禁。搜索结果可在打开前按标题暂时聚合；详情解析确认仍是普通短篇时必须折叠为当前单帖。

单个来源主题内的目录证据优先级：

1. 楼主正文中的 `misc.php?mod=tag&id=...` 作品 Tag 目录；固定论坛 HTTPS origin，以 `forcemobile=1` 从第一页开始普通 GET，最多拉取 100 页，不占用论坛搜索冷却。
2. 同帖目录中明确指向 pid 的章节链接。
3. 楼主各楼层内明确指向其他 tid 的跨帖章节链接；合并所有目录片段，不只选择单个楼层。
4. 目录上下文中的裸章节号、前后篇、分段号和卷号，例如 `1`、`5前`、`27(1)`、`190`、`04卷`。
5. 上一话/下一话明确链接形成的章节图。
6. 以上均不存在时，将主题视为独立单章来源。

Tag 目录是单个来源内的最高级跨帖目录证据，目录指定的 pid 是同帖章节正文的权威来源。只读取 Tag 页主题列表容器，拒绝跨 origin 跳转和容器外公告链接。普通回复、点评、引用块链接和无目录引用的用户楼层不进入阅读器；“小说地址”“原作小说”“漫画原作”“原作链接”等跨媒介或来源引用也不作为章节。目录链接先按 tid/pid 去重，Tag 自身去重后不足两章不能作为强目录证据。旧帖的“之一”“之十四、十五、十六”等序号统一显示为“第 1 话”“第 14、15、16 话”；同一数字话数下明确写有“其之/其の/Part/Pt”的分段使用分段标记参与去重，`第 1 话其之 1` 与 `其之 2` 必须同时保留。页码范围、投票和特典等无章节号条目按相邻明确章节插值排序，不占用整数章节号。`第 N 回……漫画祭` 等活动编号不占用作品的第 N 话；编号番外按番外序号稳定排序。Tag 中的完整帖子标题移除发布组、作者元数据和重复作品名，但保留 `2nd season`、`Another Story`、节日特典等子系列信息；楼主手写的帖内目录标签保持原文。

`6.2～6.4`、`11.1～11.2` 等没有“话/章”单位的小数范围，以及作品名后的 `02-03` 等裸整数范围，表示同一个不可再拆的合并章节：列表保留完整区间，统一显示为“第 6.2～6.4 话”“第 2～3 话”，排序取范围起点；不得只提取末尾数字，也不得在没有正文边界时展开为多个伪章节。`p.1-10` 等明确页码范围不作为章节。

漫画和小说共用一层“批量章节链接”展开规则：目录标签明确写成 `第 1-5 话/章` 等连续数字范围、链接指向其它 tid 且没有 pid 时，直接普通 GET 目标帖，不提交论坛搜索，也不递归展开目标帖中的下一层批量链接。只有目标帖能在本帖解析出范围内每一个数字章节，并且每章都有可靠 pid 正文定位时，才用实际章节替换合并范围入口；任一章缺失或目标不可拆时保留原始“第 1-5 话”入口。单楼层只含一组图片或正文、没有可靠内部边界时不得按数字范围等分伪造章节。

章节列表只在标题本身解析出可靠数字章节号时显示“第 N 话/章”；正文、番外、特典和其它无编号标题不得按列表位置伪造序号。数字序号和副标题作为同一个文本组件显示，使用同一字体、字号和颜色。

明确长篇漫画从任何入口都走同一完整解析管线：先解析作品身份并取得完整论坛搜索候选，再按楼主分组；每组选择主线章节号最高的主题作为确定性锚点，然后展开其 Tag、同帖 pid、楼内 tid 和前后话证据。活动编号、特典和番外不参与主线锚点竞争。由主动搜索进入时复用当前 searchid 补齐分页，不额外提交搜索；主页、排行或详情刷新需要新搜索时服从全局 10 秒门控。

小说热门帖子结构审计表明：轻小说/译文区 20 个样本中 19 个可在当前 tid 完成主体分章，文学区前 12 名中的 11 个实际作品帖也全部如此；跨帖主体主要是分卷单行本。小说使用独立于漫画的简化管线：

- 普通连载首次建立索引只解析当前 tid，不提交论坛搜索。即使标题带章号，也不按漫画规则搜索其它主题。
- 小说首楼若已有论坛 Tag 目录，或至少两个同 tid、不同 pid 且一一对应的章节链接，直接以普通第一页建立目录，不再为了取得正文把全部“只看楼主”分页拉完。多个目录标签共用同一 pid 时仍走完整楼主分页，以保留楼内 block 分章能力；漫画仍按完整目录片段规则合并楼主分页，不套用小说快路径。
- 已完成的完整楼主主题可以继续复用；封面后台正在执行但尚未完成的全帖请求不得阻塞目录快路径。打开带 pid 的小说章节时直接使用论坛 findpost 跳转加载目标所在页，不为阅读单章重新拉完整长帖。
- 小说版本只分为“连载版”和“单行本”。标题明确写有 WEB/连载版时属于连载版；明确写有文库/单行本，或标题只标出第 N 卷/Vol.N 而没有卷内章号时属于单行本；其它未标明版本的小说默认属于连载版。
- 当前帖出现明确卷目录，或当前主题本身明确为单行本时，才以规范作品名提交一次搜索。标题末尾只有裸 `01/02/3`，或卷号紧贴在规范作品名末尾时，先视为“卷候选”；只有同作品搜索集合内出现明确卷帖或至少两个不同候选卷号，才提升为单行本，否则仍按普通连载处理。翻页复用该次 searchid，`EX6/EX10` 等命名附篇不得提升为第六/十卷。主动搜索页必须先合并当前 searchid 中所有同作品候选，再选择确定性锚点和逐卷展开，不能因所点帖子或最新锚点不同而丢失较早分卷。
- 跨帖只追踪明确的卷链接。当前连载帖中的特典、短篇和其它关联主题不递归扩张；这类内容仍可通过原始搜索结果兜底打开。
- 每个卷帖再独立应用“同帖 pid 目录 → 过滤后的楼主正文楼层 → 单楼层可靠正文标题 → 整卷阅读兜底”。单楼超长正文只有在至少识别出两个可靠章节标题时才切分；连续目录区和正文中重复出现的标题取正文位置，章节保存同一 pid 内 0-based、前闭后开的 block 范围，阅读和下载只加载当前范围。卷内能可靠分章时不得把整卷同时作为普通章节；确实无法分章时才显示“整卷阅读”。
- 帖内目录可以有多个标签指向同一 pid。小说在“同 PID、不同目录标签”时不按 PID 提前去重；只有楼层正文中能为每个标签找到唯一、有序且包含实质正文的标题边界时，才按 block 范围拆成多章；识别不完整则保守回退。重复的番外标签可用正文标题中的 `①/②/③` 等明确分段区分，不用列表位置猜测。漫画仍按 PID 去重，没有可靠图片边界时不伪造多话。
- 连载版与单行本保存在同一作品索引中，但章节互不混排；单行本目录按卷标题分组。不同楼主始终保留独立来源。智能目录对连载版选择可靠章节覆盖最完整的来源作为基准，只从其它译者补入标题中能明确识别且基准缺失的数字章、数字范围、序章/终章/尾声/后记/幕间/番外；不得用列表位置或 `order` 猜测无编号标题。单行本按卷补缺，同一卷只选择一个楼主版本，可靠分章优先于整卷入口，不拼接同一卷内部的多个译本。
- 楼主正文序列必须排除短说明、纯插画/附件楼和非作品主题，并优先从正文中的可靠章标题取名；不得仅以 80 字阈值直接拆分。

## 10. 作品聚合

聚合以精确率优先：宁可漏合并，也不把不同作品误并。v1 不提供人工合并/拆分入口。

### 10.1 明确关系

以下关系直接建立聚合边：

- 目录链接
- 上一话/下一话链接
- 同一目录明确列出的跨帖章节

### 10.2 标题结构规则

标题处理：

- Unicode、全半角、空白、括号和 HTML 实体规范化。
- 识别位于标题前部的汉化组/翻译组、作者和发布说明。
- 识别并保留“正式连载版”“WEB版”“文库版”“单行本”等版本区分信息。小说的连载版与单行本共享作品身份，但在目录中保留版本元数据；漫画及小说的其它部/季冲突仍按独立作品处理。
- 从尾部提取 `第69话`、`102`、`8.2`、`26~34话`、`最终话`、卷彩页、附录和特典等章节标记；支持章节号直接粘在以括号、问号或波浪号结束的作品名后，也支持章节号后的中日韩副标题和前/中/后、上/下分段。
- 使用作品名、创作者签名和版本标记生成作品键；`漫画/作画` 等创作者职责写法及职责顺序变化归一为同一创作者签名，发布装饰标签不进入作者键。

规范化作品名、版本和版块/分类兼容时才允许自动聚合。双方均有创作者签名且冲突时必须拆分；一方缺少创作者签名时，只有候选集中恰好存在一个兼容的非空签名才允许归入，避免把缺作者元数据的帖子永久拆散，也避免在多个创作者候选间猜测。论坛楼主/翻译者不参与 `Work` 身份键，而用于划分独立来源目录。编辑距离或模糊相似度只能辅助判断，不能单独触发合并。

### 10.3 冲突处理

- 同一楼主目录内出现重复章节标记时，优先保留明确目录指向或确定性锚点解析出的版本。
- 详情页默认提供“智能聚合”目录：以主线章节覆盖最完整、最新话更高的楼主目录为基准；同一章节固定使用先选中的来源，不逐页混译，只从后续来源补齐基准目录缺失的章节。
- 多来源目录左上角直接显示当前目录名称、章数和下拉箭头，不在“智能聚合”左侧重复显示漏斗图标；点击整块文字仍打开来源筛选菜单。
- 跨来源去重不得删除同一楼主目录内同序号的不同 pid、特别篇或其它独立条目。各楼主独立目录始终完整保留，作为智能聚合失败时的一级兜底。
- 多目录作品中的链接无法可靠归属楼主时，不猜测归并，保留在独立的“未归属来源”目录。
- 小说的“连载版/WEB版”和“单行本/文库版”不拆成不同作品，而是在同一作品内形成互不混排的版本筛选；其它版本标记冲突时拆成不同作品。
- 小说在连载版和单行本各自范围内生成保守的智能补缺目录；只有智能结果实际使用至少两个独立来源时才显示“智能聚合”。其它楼主仍通过同一个来源/版本筛选按钮完整切换，原始结果继续作为最终兜底。
- 聚合失败只影响作品列表便利性，不影响单帖打开和阅读。

## 11. 封面

封面采用与作品目录分离的两阶段持久模型：

- 索引建立前以 `LibraryKind + tid` 保存临时封面；索引完成后以 `LibraryKind + workId` 保存固定正式封面，并将作品全部来源 tid 映射到同一记录。漫画与小说始终按 `LibraryKind` 隔离。
- 正式封面成功后不因来源集合扩张、目录切换、索引刷新或页面重建失效。只有清缓存或用户长按详情封面选择“重新解析封面”才重新选择。
- 正式候选最多检查三个：旧正式封面、根帖/目录帖、漫画最早主线章节或小说连载根帖/单行本第一卷，最后以当前入口帖补足。封面模块不自行搜索作品或重复解析目录。
- 小说首楼正文中的合格图片可直接作为封面；后续楼主楼层的图片只有在紧邻文本或图片说明明确包含“封面/书影/封绘/彩插/插图/cover/illustration”时才可入选。“图文无关/表情/截图/晒图/闲聊/梗图/聊天/吐槽/感想”等否决语义优先，无高置信候选时固定使用文字封面。漫画继续使用既有正文图片规则。
- 临时封面在正式图片下载、解码和原子落盘完成前一直保留，成功后以 180 ms 淡入替换。列表和详情都不显示封面转圈。
- “确认无图”固定显示文字封面；网络、权限、下载或解码失败使用 5 分钟、1 小时、24 小时封顶的可重试退避，且任何失败都不删除当前有效图片。
- 图片需可被 Flutter 完整解码且宽高均大于 96 px，继续排除头像、表情、论坛 Logo、签名和小装饰图。
- 封面有效性检查读取原始尺寸后只解码最长边不超过 256 px 的验证帧，不为验证分配完整像素缓冲。原始封面文件继续持久保存；列表、网格和详情按实际物理显示高度请求等比解码尺寸，避免缩略图占满 Flutter 图片缓存，同时不改变封面选择、缓存键和磁盘清晰度。
- 封面请求由实际显示需求持有；顶层隐藏入口和小说隐藏子分区不得启动封面请求，最后一个可见需求离开后取消尚未完成的非主动任务，取消不得写入失败缓存。用户主动重新解析不受单个卡片离屏影响。
- 触摸或手写笔按下后暂停新的封面解析和后续重阶段；手指离开但惯性滚动仍在继续时保持暂停，滚动结束或连续 500 ms 无滚动通知后恢复。已经发出的 HTTP 请求允许完成，但必须在帖子请求、图片下载、解码与落盘之间重新检查暂停和需求状态。
- 自动封面在普通第一页无候选时最多补探测 1 页“只看楼主”，不得为寻找封面遍历完整长帖。详情建立或主动刷新作品索引期间暂停封面队列，目录请求完成后再恢复，避免非关键图片探测抢占核心阅读路径。
- 无图时使用统一中性浅灰蓝/深灰背景、低对比类型图标和标题；不使用哈希彩色渐变、阴影或装饰字样。

## 12. 搜索

搜索使用论坛原生高级搜索，不预抓或建立全论坛索引；只为用户实际打开的作品建立本机持久化目录索引。

- 漫画限制 srchfid[]=30。
- 小说同时限制 srchfid[]=49 和 srchfid[]=55。
- 用户点击搜索按钮或键盘提交后才发起新搜索，不做输入即搜索。
- 论坛规定 10 秒内只能进行一次新搜索。提交后按钮显示倒计时并禁用，倒计时以最后一次服务器接受的搜索为准。
- 翻页复用论坛返回的 searchid，不视为新搜索，也不重新提交关键词。
- 解析 searchid、结果分页、标题、摘要、来源和主题信息。
- 搜索结果默认显示智能聚合作品，并通过页面底部等宽纯文字栏切换到未经聚合的“原始结果”，模式栏不显示图标；选中项使用主题主色，未选中项在浅色模式使用深色文字、深色模式使用浅色文字。界面不再显示“最终兜底”字样。搜索页缺少分类字段时，标题结构聚合只作为候选展示；点击后必须由统一协调器读取真实分类和主题目录证据再确认，不能让搜索入口拥有不同于主页的最终分类规则。原始结果直接打开对应 tid，不建立或刷新作品索引，详情页明确提示“未进行作品聚合，目录可能不完整”。
- 搜索框右侧的进行中状态使用固定正方形约束的圆形加载指示器，不得随输入框后缀区域压缩为椭圆。
- 详情页快捷搜索使用当前解析出的规范作品名，默认进入原始结果。先读取完全匹配的本机搜索缓存；无缓存时再发起论坛搜索，并继续服从全局 10 秒门控，冷却未结束时在页面内等待后自动执行。
- 标题开头的单个未知括号项后若仍有完整作品名和明确章节号，该括号项按作者元数据处理；括号后只有章节号时，括号内容仍是作品名。发布组标签位于作者括号前后不得改变作品身份，也不得限制不同楼主之间的兼容聚合。
- 缓存近期搜索结果用于返回页面和短期离线回看，但不把缓存伪装成完整搜索。
- 频率限制、验证码、searchid 过期和连接重置分别显示明确状态。

### 12.1 按需作品索引

- 漫画和小说共用同一套全局作品索引存储，不按论坛账号分区；索引只保存标题、作者、分类、tid/pid、章节顺序和来源 URL，不保存帖子正文或摘要。
- 作品键、来源 tid 映射查询和作品 ID 查询始终带有 `LibraryKind`；漫画与小说即使标题完全相同，或某个目录误收录了另一类型的 tid，也不得命中、覆盖或删除对方索引。
- 主页、排行、搜索和详情刷新统一调用同一个作品解析协调器；入口只提供当前来源 tid 和可复用的搜索上下文，不各自拼装目录。
- 首次点击先按 tid 和兼容作品键查本地索引。有效索引命中时直接打开；未命中时加载当前主题、识别真实分类并解析主题内目录证据。明确短篇默认保持单帖，仅允许同作品的强结构化目录证据纠正误标。
- 漫画只有分类为 `#長篇連載/#长篇连载`、`#韩国漫画`、`#泰国漫画`、`#欧美其他`（含繁体）时才允许自动跨帖搜索；分类名称和论坛 typeId 双重识别。非长篇漫画即使标题带数字也不自动搜索。
- `#短篇漫畫/#短篇漫画` 使用分类名称和真实 typeId 68 双重识别。未发现强目录证据时严格作品键按 tid 隔离，不能因连续章节号、短篇集链接或已有同名索引互相合并；发现强证据并提升为长篇后，搜索候选、确定性锚点和全部来源 tid 使用与明确长篇相同的解析及持久化规则。
- 明确长篇漫画在索引未命中、主动搜索或详情下拉刷新时先完整取得搜索候选，再按楼主分别选取最高主线章节锚点并解析目录；已有 Tag 不能跳过这一步。小说普通连载只解析当前 tid；当前主题或明确目录表明存在单行本卷时允许搜索。裸 `01/02/3` 或紧贴作品名的卷号候选也允许一次确认搜索，但必须由结果中的明确卷标记或至少两个不同候选卷号相互印证；成立后先合并当前 searchid 内所有兼容译者和分卷，再逐卷解析。
- 自动建立或详情刷新需要新搜索时共用全局 10 秒门控；主动搜索对明确长篇结果复用当前 searchid 拉完分页，不进行第二次 POST，也不重复消耗一次搜索。
- 已索引作品从主页直接打开。主页遇到严格作品键相同但未收录的新 tid 时只提示可能更新，不自动重搜；用户再次主动搜索或在详情下拉才刷新。
- 刷新先完整取得候选结果，再与旧索引做不降级合并并原子提交；任一分页失败时保留旧索引和更新时间。
- 同一楼主目录内的同序号条目按确定性规则更新；不同楼主目录互不去重。预告、特典或衍生作品没有充分身份信息时保守保留。
- 所有来源主题和章节 tid 均映射到同一作品，因此从任一入口进入都得到相同的目录集合；多来源作品默认选中智能聚合目录，当前入口 tid 只用于定位来源和原帖，不改变默认聚合结果。
- 索引记录携带解析版本。解析规则升级时旧记录保留在数据库中但不再命中，首次打开自动重建并原子覆盖；阅读历史、下载、收藏和临时缓存不受影响。

## 13. 云端收藏

论坛云端帖子收藏是唯一收藏数据源，本地不提供另一套收藏。

- 读取 `home.php?mod=space&do=favorite&view=me&type=thread` 及分页。
- 解析 tid、favid 和标题。
- 添加收藏前获取最新 formhash，向论坛收藏表单提交 tid。
- 取消收藏使用对应 favid，并在用户操作后提交论坛确认表单。
- 写操作以服务器成功响应为准，再更新本地缓存；失败时回滚 UI。
- 离线只显示缓存，不排队执行收藏或取消收藏。
- 只把 fid=30、49、55 的收藏映射到 APP；其他版块收藏不显示、不删除、不修改。
- 漫画收藏和小说收藏各自在页面底部提供等宽纯文字“智能聚合 / 原始收藏”切换，不显示图标，并与搜索模式栏使用相同的明暗主题文字颜色。漫画原始收藏只显示 fid=30，小说原始收藏只显示 fid=49/55；两类数据不混排。原始收藏逐条显示论坛标题，点击后按原始帖子进入且不建立作品索引。

论坛收藏单位是帖子，APP 收藏单位是作品：

- 未聚合的原始结果按单帖收藏，只收藏当前 tid。
- 聚合作品按作品收藏，只向论坛提交一个确定性锚点。该锚点直接复用默认目录解析时选择的基准帖：优先主线章节，随后按最高主线章节号、发帖时间和 tid 稳定决胜；不另行猜测“专门目录帖”。
- 详情页在两种语义下都只显示“收藏”，由图标状态表达已收藏/未收藏，不把内部收藏单位写进按钮。
- 读取收藏后按作品聚合规则去重，多个已收藏章节只显示一个作品。
- 作品的收藏状态为“全部来源帖、目录来源 tid 和章节来源 tid 中至少一个存在于云端收藏”。从不同入口或不同来源目录进入同一作品必须得到相同状态。
- 取消作品收藏删除该作品全部已映射的论坛收藏记录；存在多条时先显示记录数量并一次确认。未识别帖子和其他版块记录不删除、不修改。
- 原始收藏中取消操作只删除当前 favid；智能聚合模式继续按作品删除全部已映射记录。离线缓存同时支持两种视图，但均保持只读。

## 20. 技术架构

采用新建现代 Flutter 项目，不直接在旧项目上替换 API。

建议分层：

```text
lib/
    app/                 应用入口、主题、路由、响应式外壳
    core/
        network/         会话、Cookie、重试、限流和认证媒体
        storage/         数据库、安全存储、缓存和文件
        parsing/         通用 HTML/Discuz 解析工具
    features/
        auth/
        comic/
        novel/
        catalog/
        work/
        search/
        favorites/
        history/
        downloads/
        settings/
    shared/              复用模型、组件和状态视图
```

技术选择原则：

- 使用当前 Flutter stable 和 Dart 3。
- 使用 feature-first 目录，解析、存储和 UI 不互相直接依赖实现细节。
- 网络层使用支持拦截器与 CookieJar 的客户端。
- HTML 使用 Dart DOM 解析库，不使用正则表达式解析完整 HTML。
- 关系数据和任务状态使用三端均可用的 SQLite 持久层。
- 凭据使用平台安全存储。
- 状态管理采用无代码生成或少代码生成的明确单向数据流方案。
- 路由支持窄屏全页跳转和宽屏嵌套详情导航。
- 论坛图片通过认证媒体仓库下载和缓存，不直接把敏感 Cookie 拼到 Widget 层。
- WebView 代码使用平台条件实现，Linux 构建不引用 WebView 平台库。

依赖版本在实现开始时选择当时最新、维护活跃且同时支持目标平台的稳定版本，锁定到 `pubspec.lock`。

## 21. UI 复刻与响应式规则

- 以参考项目源码和 `document/screenshot_light.jpg`、`screenshot_dark.jpg` 为视觉基线。
- 显式定义主题 token，不依赖新版 Flutter 默认 Material 主题，以避免 Material 版本变化导致外观漂移。

- Android/iOS 处理安全区、系统状态栏、返回手势和沉浸式阅读。
- 明暗主题必须覆盖列表、卡片、弹窗、阅读器、下载页和错误状态，不允许局部硬编码浅色。

手写 Dart 代码遵守项目 AGENTS.md：4 空格缩进，大括号换行。生成的平台工程文件保持 Flutter 工具生成格式，不进行无关重排。

## 22. 缓存和本地存储

- 凭据：安全存储。
- Cookie：账号隔离的持久 CookieJar。
- 设置：轻量键值存储或本地数据库设置表。
- 作品/章节索引、收藏缓存、历史和下载任务：SQLite。
- 封面使用独立作品级缓存；在线阅读正文媒体使用应用临时文件缓存；已完成下载保存在下载支持目录。
- 已完成下载、作品索引、封面和在线正文临时缓存分开管理。临时缓存、封面缓存和作品索引分别提供带二次确认的清理入口，任何一种清理都不删除阅读历史或已完成下载；退出登录同时清理临时缓存和封面缓存，作品索引继续保留。
- 每个 App 进程仅在首次进入已登录主页、首帧结束约 5 秒后执行一次自动维护，不在阅读、下载或页面滚动过程中再次触发容量扫描。在线正文图片超过 1 GB 时按最近使用时间由旧到新清理至 0.5 GB；封面超过 256 MB 时清理至 128 MB。缓存命中最多每天更新一次文件时间，清理同时删除零字节文件、超过一天的残留分片和无数据库引用的旧封面文件。
- 搜索缓存由同一次启动维护按漫画、小说分别保留最近 50 个关键词；云收藏缓存继续在同步时整体覆盖。
- 所有缓存记录保存来源 URL、来源 ID、更新时间和文件状态，删除异常文件后可自动修复索引。

## 23. 错误与状态设计

需要独立区分：

- 未登录
- 凭据错误
- 需要验证码
- 验证码错误/过期
- 会话过期
- 权限不足或内容隐藏
- 搜索 10 秒冷却
- 搜索 searchid 过期
- 论坛限流
- TLS/连接重置
- HTML 模板变化/解析失败
- 图片或附件不存在
- 本地空间不足
- 下载被网络策略暂停

所有页面复刻参考应用的加载、空状态和错误状态风格。错误信息使用用户能理解的中文，不显示堆栈、Cookie 或完整请求参数。

## 28. 已知风险

- 论坛 HTML 模板变化会导致解析器失效；通过集中选择器、合成 fixture 和原帖兜底降低影响。
- 论坛 TLS 重置和限流可能造成短时失败；通过低并发、缓存和幂等退避处理。
- 标题写法不统一会造成漏合并；这是精确率优先策略的已接受结果。
- 跨帖作品没有可收藏的独立目录主题时，云端收藏使用默认目录的确定性解析基准帖作为单一代表锚点。
- 列表无封面导致首次可见卡片需要额外请求；通过惰性低并发和持久缓存控制负载。
- iOS 无法在当前 Linux 环境完成构建和实机验证，正式发布前必须在 Apple 工具链补验。
- 论坛权限和回复可见内容不会被绕过，部分作品可能只能通过原帖查看。
